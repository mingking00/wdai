# 反思：Agentic Patterns分析过程

## 做了什么

1. **发现资源**：找到《Agentic Design Patterns》PDF + 中文翻译
2. **差距分析**：对比21种模式 vs WDai当前实现
3. **制定计划**：划分P0/P1/P2，规划evo-006到evo-017
4. **标准化**：创建文档规范，更新evo任务格式

---

## 问题识别

### 问题1：信息获取不完整
**症状**：只获取了PDF目录，没有深入各章节内容
**后果**：
- ReAct/ToT的具体实现细节未知
- MCP协议的具体规范未读
- Token估算是拍脑袋，不是基于实际复杂度

**应该**：先深入读Chapter 6和10，再制定计划

### 问题2：计划先于验证
**症状**：直接创建evo-006/007/008文档
**后果**：
- 未验证Planning是否真是当前最大瓶颈
- 未测试现有系统在处理规划任务时的实际表现
- 可能在做"我觉得需要"而不是"真的需要"

**应该**：先跑几个测试用例，证明Planning缺失确实影响性能

### 问题3：文档过度工程
**症状**：花大量时间做YAML frontmatter标准化
**后果**：
- 1.5小时花在格式上
- 核心问题（如何实现Planning）还没想清楚
- 形式主义优先于实质内容

**应该**：先用最简格式记录，价值验证后再标准化

### 问题4：缺乏用户视角
**症状**：按书中章节顺序排优先级
**后果**：
- P0选了Planning/MCP/Reasoning
- 但没问：用户当前最痛的是什么？
- 书中P0不等于用户的P0

**应该**：先问"你现在最卡在哪里？"，再对答案做模式匹配

### 问题5：并行线头太多
**症状**：同时做分析+标准化+创建evo文档
**后果**：
- 注意力分散
- 每个都只做了一半深度
- 没有单一交付物是完整的

**应该**：单线程推进，完成一个再下一个

  - 确认Planning缺失确实是瓶颈

### 短期改进（本周）

- [ ] **建立"先问再猜"规则**
  - 任何P0决策前，必须有用户确认或实测数据
  - 禁止"我觉得你需要X"
  - 模板："检测到可能缺口X，需要验证吗？"

  - 单任务推进，禁止并行线头

  - 未完成不准开下一个

- [ ] **创建快速验证协议**
  - 任何新功能先做最小可行版本（MVP）
  - MVP = 能跑一个用例，代码<100行
  - 验证价值后再完整实现

### 长期改进（持续）

- [ ] **反形式主义检查**
  - 每次花>30分钟在文档/格式上时，触发警报
  - 自问：这部分格式对用户价值是什么？
  - 如无直接价值，回退到最简

- [ ] **用户痛点优先原则**
  - 建立"痛点→模式→方案"映射
  - 任何方案必须追溯到具体痛点
  - 书中模式只是工具箱，不是todo list

- [ ] **度量驱动改进**
  - 每个evo必须有可度量目标
  - 启动前：基线数据
  - 完成后：对比数据
  - 无数据 = 未完成

## 核心教训

1. **书是参考，不是指令**：21种模式是工具箱，不是todo list
2. **痛点驱动，而非模式驱动**：先找痛，再匹配工具
3. **深度>广度**：一个跑通的demo > 十个规划文档
4. **形式服务于内容**：文档格式是手段，不是目的
5. **验证>假设**：实测数据 > 拍脑袋判断
