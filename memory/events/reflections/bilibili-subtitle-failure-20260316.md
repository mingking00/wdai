## 二、根本原因分析

### 1. 方法选择失误（P1 - 元能力层）
**问题**: 同一方法失败3次才换路  
**违反原则**: 创新能力 - "3次失败强制换路"  
**应该**: 第1次412错误后立即换API方案

### 2. 缺乏实时持久化（P2 - 执行策略层）
**问题**: 转录结果保存在/tmp/临时目录  
**违反原则**: 检查验证 - 未确保结果持久化  
**应该**: 每完成一段立即写入 workspace/memory/

### 3. 忽视系统约束（P0 - 安全层）
**问题**: 未考虑系统对长进程的自动清理  
**违反原则**: 物理现实检查 - 系统资源限制  
**应该**: 预估转录时间，采用更短的分段或并行处理

### 4. 反思Agent失职（P1 - 元能力层）
**问题**: 失败后仅记录"还在处理"，未立即分析  
**违反原则**: 双路径认知 - 死循环检测失效  
**应该**: 检测到长时间无进展立即触发反思

## 三、提炼原则

### 原则1：临时目录禁令 ⭐
```
任何需要保留的结果，禁止保存在/tmp/或/var/tmp/。
必须写入：workspace/、memory/、或用户指定目录。
执行前检查："这个路径重启后还在吗？"
```

### 原则2：实时持久化协议 ⭐
```
长时间任务必须：
1. 每完成一个子任务立即保存结果
2. 使用append模式追加到文件，而非最后一次性写入
3. 保存路径必须在workspace/下
4. 任务中断后可从断点恢复
```

### 原则3：进程生命周期检查 ⭐
```
启动可能超过10分钟的任务前：
1. 评估总耗时 = 任务数 × 单任务耗时
2. 如果>15分钟，必须：
   - 缩短分段（如5分钟→2分钟）
   - 或并行处理
   - 或换更快的方法（如换smaller模型）
```

### 原则4：失败快速止损 ⭐
```
同一方法失败时：
- 第1次：记录，retry一次
- 第2次：记录，检查参数/环境
- 第3次：立即锁定该方法，强制换路
禁止：第4次仍用同一方法
```
