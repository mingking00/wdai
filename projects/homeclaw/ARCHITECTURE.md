## 关键设计决策

### 1. 为什么分层？

| 层级 | 职责 | 技术选型 | 安全边界 |
|------|------|----------|----------|
| **感知层** | 数据采集 | 传感器硬件 | 只输出数据，不联网 |
| **边缘层** | 本地AI推理 | 树莓派5/Jetson Nano | 不外出数据，只发指令 |
| **云端层** | 执行+交互 | VPS/Home Server | 接收指令，控制设备 |

**安全优势**: 敏感的位置、行为数据永远不出本地网络

### 2. 蓝牙Beacon识别可行性

```yaml
技术方案:
  硬件:
    - 信标: 手环/胸牌/手机 (iBeacon/Eddystone)
    - 接收器: ESP32/树莓派 (每个区域1个)
    
  定位精度:
    - 单点: 房间级 (3-5米)
    - 多点三角定位: 区域级 (1-2米)
    
  身份绑定:
    - Beacon UUID ↔ 用户ID 映射表
    - 支持多人、权限分级
    
  功耗:
    - 手环: 1年续航 (CR2032)
    - 接收器: USB供电

工程可行性: ★★★★★
成本: 低 (< ¥50/人)
```

### 3. 毫米波雷达选择

```yaml
推荐方案:
  型号: LD2410B / LD2450 (海凌科)
  价格: ¥30-80
  功能:
    - 人体存在检测 (无动作也能检测)
    - 距离测量 (0.4-8米)
    - 微动检测 (呼吸级)
    - 多人检测 (LD2450支持3人)
    
  优势:
    - 不侵犯隐私 (无摄像头)
    - 不受光照影响
    - 可穿透非金属材料
    
  精度:
    - 距离: ±0.1米
    - 角度: ±5度
    - 足以判断"在书桌前" vs "只是路过"
```

### 4. 本地AI模型选择

```yaml
方案对比:

方案A: 规则引擎 (推荐起步)
  - if 人在书桌区域 AND 光照<300lux AND 静止>3s: 开灯
  - 优点: 可解释、无延迟、易调试
  - 缺点: 不够智能
  
方案B: TinyML模型 (推荐进阶)
  - 输入: [人员ID, x, y, 光照, 时间, 上次行为]
  - 模型: 轻量级决策树/神经网络 (< 1MB)
  - 平台: TensorFlow Lite / ONNX Runtime
  - 推理时间: < 10ms
  - 优点: 可学习用户习惯
  
方案C: 本地LLM (高端方案)
  - 模型: Phi-3 / Llama-3-8B (量化版)
  - 要求: 8GB+ RAM
  - 优点: 自然语言理解
  - 缺点: 成本高、延迟大

推荐路径: A → B → C (渐进式升级)
```

## 改进建议

### 你的原始架构问题与改进

| 原始设计 | 潜在问题 | 改进方案 |
|---------|----------|----------|
| 单一雷达判断 | 误判率高（路过 vs 停留） | 多传感器融合 + 时间阈值 |
| 边缘AI直接控制 | 网络中断时失控 | 边缘层保留本地应急规则 |
| 蓝牙单独识别 | 信号漂移、多房间干扰 | 蓝牙+雷达联合定位 |
| 无反馈机制 | 用户被"强制"控制 | 飞书确认 + 学习模式 |

### 关键改进点

#### 1. 感知融合算法

```python
# 伪代码：多传感器融合
class PresenceDetector:
    def detect_intent(self, beacon_data, radar_data, env_data):
        # 1. 身份确认
        person = self.identify_by_beacon(beacon_data)
        if not person:
            return None  # 不认识的人，不响应
        
        # 2. 位置确认 (雷达为主，蓝牙校准)
        position = self.fuse_position(
            radar_coords=radar_data.coords,  # 精确但可能丢帧
            beacon_zone=beacon_data.zone     # 粗但稳定
        )
        
        # 3. 行为判断
        if radar_data.is_moving:
            activity = "moving"  # 路过
        elif radar_data.stationary_duration > 3:
            activity = "working"  # 停留工作
        else:
            activity = "unknown"
        
        # 4. 环境判断
        needs_light = env_data.lux < person.preferred_lux
        
        # 5. 时间/场景判断
        is_work_hours = 9 <= now.hour <= 18
        
        # 6. 综合决策
        if activity == "working" and needs_light and is_work_hours:
            return Intent("turn_on_light", confidence=0.95)
        
        return None
```

#### 2. 置信度分级响应

```yaml
响应策略:
  高置信度 (> 0.9):
    action: 立即执行
    notify: 飞书静默通知 (可关闭)
    
  中置信度 (0.7-0.9):
    action: 预执行 (延迟3秒)
    notify: 飞书"即将开灯，取消请回复N"
    
  低置信度 (< 0.7):
    action: 询问用户
    notify: 飞书"检测到您在书桌前，需要开灯吗？"
    
  紧急场景 (安防/夜起):
    action: 立即执行
    notify: 飞书提醒 + 本地蜂鸣器确认
```

#### 3. 防误触设计

```python
# 常见误触场景及解决方案

误触1: 路过书桌
  解决: 静止检测 > 3秒才触发

误触2: 打扫房间 (不需要开灯)
  解决: 运动模式识别 (大动作=打扫，小动作=工作)

误触3: 已经开灯但人离开
  解决: 人走灯灭延迟 (5分钟无检测才关闭)

误触4: 多人场景
  解决: 权限优先级 (主人 > 访客)

误触5: 白天不需要灯
  解决: 光照传感器阈值 (< 300 lux)
```

## 数据流示例

### 场景：用户走到书桌前工作

```
时间轴:
T+0s   [蓝牙] 检测到用户A进入房间
       [雷达] 检测到人体，坐标(3.2, 1.5)，移动中
       
T+1s   [边缘AI] 融合数据 → 用户A在移动中，置信度低，不动作

T+3s   [雷达] 用户停止移动，坐标(3.1, 1.4)，静止
       [光感] 光照 250 lux (< 阈值300)
       
T+3.1s [边缘AI] 推理:
       - 身份: 用户A (权限: 可控制书桌灯)
       - 位置: 书桌区域 (置信度0.95)
       - 状态: 静止3秒 (置信度0.98)
       - 环境: 需照明 (置信度0.90)
       - 时间: 工作时段 (14:30)
       → 综合置信度: 0.94
       
T+3.2s [边缘AI] 生成指令:
       {"action": "turn_on", "device": "desk_lamp", "brightness": 80}
       
T+3.3s [云端] HomeClaw接收指令 → 调用HA API → 灯亮

T+3.5s [飞书] 静默通知: "💡 已为您打开书桌灯 (亮度80%)"

T+5min [雷达] 检测到用户离开书桌区域
T+10min[雷达] 书桌区域无人 > 5分钟
T+10min[边缘AI] 生成关闭指令 → 灯灭
```

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 蓝牙信号干扰 | 身份识别失败 | 多基站冗余 + 雷达辅助 |
| 雷达误判 | 误开/误关灯 | 时间阈值 + 置信度分级 |
| 网络中断 | 无法控制 | 边缘层本地应急规则 |
| 隐私担忧 | 用户抵触 | 本地处理，数据不出屋 |
| 学习期体验差 | 频繁误触 | 初始使用确认模式，逐步自动化 |

## 总结

这个架构是**工程上完全可行**的，而且是一个**先进的隐私优先设计**。

核心优势:
1. ✅ 本地感知，隐私安全
2. ✅ 多传感器融合，准确率高
3. ✅ 分层架构，可扩展
4. ✅ 渐进智能，体验可控

建议从**规则引擎起步**，验证场景后再引入ML。
