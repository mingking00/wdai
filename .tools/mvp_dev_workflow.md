# 渐进式MVP开发流程

> 从简单开始，让真实需求驱动架构演进

---

## 核心原则

> **"好架构是被逼出来的，不是设计出来的"**

不要预先设计完美架构，从能跑通的最小版本开始，让真实使用中的痛点驱动改进。

---

## 三阶段模型

### Phase 1: 骨架阶段 - "能跑通"

**目标**: 一条主流程能跑通，证明概念可行

**时间盒**: 2-4小时

**成功标准**:
- [ ] 核心功能可用（哪怕很粗糙）
- [ ] 没有明显的阻塞性bug
- [ ] 你自己能用它完成一次完整流程

**不要做的事**:
- ❌ 多用户支持
- ❌ 错误处理
- ❌ 性能优化
- ❌ 配置选项
- ❌ 文档

**示例 (Vibero阅读器 Phase 1)**:
- Flask后端 + 单页面
- 只能上传一个txt文件
- 只翻译当前章节
- 术语提取很粗糙
- 但**能用**

---

### Phase 2: 反馈阶段 - "能用吗？"

**目标**: 给用户使用，收集真实反馈

**时间盒**: 1-3天（取决于用户反馈速度）

**关键动作**:
1. 给1-3个用户试用
2. 观察他们怎么用（不是听他们说，是看他们做）
3. 记录所有卡顿点和意外用法
4. 区分：高频痛点 vs 低频需求

**反馈收集模板**:
```
用户: [谁]
场景: [用来做什么]

卡顿点:
- [ ] 在哪里卡住了？
- [ ] 花了多长时间？
- [ ] 他们怎么解决的？

意外用法:
- [ ] 他们做了什么你没预料到的？
- [ ] 这说明什么需求？

高频 vs 低频:
- 高频（必须修）: [ ]
- 低频（可延后）: [ ]
```

**示例 (Vibero阅读器 Phase 2)**:
- 用户反馈: "只能翻译一章，我想看整本书"
- 发现需求: 整书翻译 + 进度显示
- 高频痛点: 手动点每个章节太麻烦

---

### Phase 3: 演进阶段 - "更好用"

**目标**: 根据反馈选择性增强，解决真实痛点

**时间盒**: 每个增强点2-4小时

**选择增强点的原则**:
1. 解决最高频的痛点
2. 不增加复杂度（除非必要）
3. 保持主流程简洁

**演进路径示例**:

```
Phase 1: 单章节翻译 ✓
    ↓ 用户反馈：想翻译整本书
Phase 2: 整书批量翻译 ✓
    ↓ 用户反馈：不想配置服务器
Phase 3: 无服务器方案 ✓
    ↓ 用户反馈：...
Phase 4: [下一个真实痛点]
```

**不要做的事**:
- ❌ 同时做多个增强
- ❌ 做用户没要求的"锦上添花"
- ❌ 重构整个架构（除非痛点证明必须）

---

## 决策检查点

每个Phase结束时，必须回答：

### Phase 1 → Phase 2
- [ ] 核心功能验证了吗？
- [ ] 有愿意试用的用户吗？
- [ ] 准备好接受批评了吗？

### Phase 2 → Phase 3
- [ ] 收集了多少反馈？
- [ ] 最高频痛点是什么？
- [ ] 解决它需要重构吗？

### 继续演进 vs 停止
- [ ] 还有高频痛点吗？
- [ ] 边际收益 > 边际成本吗？
- [ ] 用户还在用吗？

---

## 反模式警示

### 反模式1: 完美主义延迟
❌ "等我把XX功能做好再发布"  
✅ 现在能跑通就发布，边用边改

### 反模式2: 架构先行
❌ 先设计好4个Phase再开始  
✅ 只设计Phase 1，后面看需求

### 反模式3: 忽视反馈
❌ "用户说的不对，我觉得应该..."  
✅ 观察用户实际行为，数据说话

### 反模式4: 过早优化
❌ Phase 1就做性能优化  
✅ 等性能成为痛点再优化

## 与Claude Code架构的对比

| 阶段 | Claude Code演进 | 我们的MVP对应 |
|------|-----------------|---------------|
| v0 | 单体循环 | Phase 1: 核心功能 |
| v1 | + To-do List | Phase 2: 用户反馈"任务管理混乱" |
| v2 | + Sub-agents | Phase 3: 长对话模型变笨 |
| v3 | + Skills | Phase 3: 系统提示过长 |
| v4 | + Async Buffer | Phase 3: IO阻塞问题 |

**关键洞察**: 每个改进都是解决前一个版本的真实痛点

---

## 应用检查清单

开始新项目前：
- [ ] Phase 1目标是否足够小（2-4小时）？
- [ ] 成功标准是否明确？
- [ ] 找到试用的用户了吗？

Phase 1完成后：
- [ ] 是否 resist 了添加更多功能的冲动？
- [ ] 是否立即给用户试用？

Phase 2进行中：
- [ ] 是否在观察而非询问？
- [ ] 是否记录了所有卡顿点？

Phase 3进行中：
- [ ] 是否只解决最高频痛点？
- [ ] 是否避免重构整个架构？

## 示例: Vibero阅读器演进

```
用户: "想要双语阅读器"

Phase 1 (3小时):
- Flask后端
- 上传txt → 分析 → 翻译当前章节
- 简陋但可用

用户试用反馈:
- "每次只能看一章，想看整本书"
- "术语要自己查，很麻烦"

Phase 2 (4小时):
- 整书批量翻译 API
- 术语自动提取
- 进度显示

用户试用反馈:
- "很好！但必须开服务器，能直接生成HTML吗？"

Phase 3 (3小时):
- novel_processor.py - 无服务器方案
- 生成自包含HTML
- 云端API + 本地模型双版本

下一步:
- 等待用户使用反馈...
```
