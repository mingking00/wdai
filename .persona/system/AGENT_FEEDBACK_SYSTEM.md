> 四个Agent的闭环反馈 + 大循环机制设计

## 🔄 每个Agent的反馈机制

### 1. SRA (Self-Reflection Agent) - 复盘Agent

**核心职责**: 复盘分析、技巧提取、洞察发现

**输入反馈**:
| 来源 | 反馈类型 | 用途 |
|------|---------|------|
| 用户 | "这次复盘有用吗？" | 验证复盘质量 |
| CodeDev | 开发任务结果 | 验证复盘建议的有效性 |
| SEA | 系统改进效果 | 验证进化建议的可行性 |
| 自身 | 技巧使用频率 | 评估提取技巧的价值 |

**输出反馈**:
- 向 IER-KG 写入: 复盘方法经验、技巧模式、洞察模式
- 向 CodeDev 推送: "下次类似任务建议"
- 向 SEA 推送: "需要改进的系统性问题"

**反馈循环**:
```
SRA完成复盘 → 生成建议 → CodeDev/SEA执行 → 结果返回 → SRA验证
                ↓                                        ↑
              写入IER-KG (追踪建议成功率) ←─────────────┘
```

**关键指标**:
- 复盘建议采纳率
- 采纳后任务成功率
- 技巧复用率
- 洞察准确性

---

### 2. SEA (System Evolution Agent) - 系统进化Agent

**核心职责**: 代码改进、重构、架构优化

**输入反馈**:
| 来源 | 反馈类型 | 用途 |
|------|---------|------|
| 用户 | 改进是否满意？ | 验证改进质量 |
| SRA | 系统性问题分析 | 确定改进方向 |
| CodeDev | 代码执行结果 | 验证改进有效性 |
| 自身 | 改进后Bug率 | 评估改进质量 |

**输出反馈**:
- 向 IER-KG 写入: 重构模式、设计决策、性能优化经验
- 向 CodeDev 推送: "代码规范更新"
- 向 SRA 推送: "改进后的效果数据"

**反馈循环**:
```
SEA完成改进 → 部署代码 → CodeDev使用 → 问题反馈 → SEA迭代
                 ↓                              ↑
               记录改进效果 (成功率/性能提升) ──┘
```

**关键指标**:
- 改进成功率
- 改进后性能提升
- 回滚率
- 技术债务减少量

---

### 3. CodeDev Agent - 代码开发Agent

**核心职责**: 实际编码、测试、交付

**输入反馈**:
| 来源 | 反馈类型 | 用途 |
|------|---------|------|
| 用户 | 代码是否可用？ | 验证交付质量 |
| SRA | 复盘建议 | 优化开发流程 |
| SEA | 代码规范/工具更新 | 使用最新最佳实践 |
| IER-KG | 相关经验 | 避免重复踩坑 |
| 自身 | Bug率/测试通过率 | 评估代码质量 |

**输出反馈**:
- 向 IER-KG 写入: 开发模式、工具技巧、优化经验
- 向 SRA 推送: "本次开发的问题/收获"
- 向 SEA 推送: "发现的系统性问题"

**反馈循环**:
```
CodeDev完成任务 → 生成报告 → SRA复盘 → 提炼建议 → CodeDev应用
                      ↓                      ↑
                    写入IER-KG (追踪经验有效性) ┘
```

**关键指标**:
- 任务完成率
- 代码Bug率
- 测试通过率
- 开发效率趋势

---

### 4. IER-KG - 经验知识图谱

**核心职责**: 经验存储、检索、验证、淘汰

**输入反馈**:
| 来源 | 反馈类型 | 用途 |
|------|---------|------|
| SRA | 复盘经验 | 更新反思类经验 |
| SEA | 改进经验 | 更新代码类经验 |
| CodeDev | 开发经验 | 更新实践类经验 |
| 自身 | 经验使用频率/成功率 | 评估经验价值 |

**输出反馈**:
- 向所有Agent: 相关经验检索结果
- 向反馈Hub: 经验有效性统计
- 向系统: 低质量经验淘汰建议

**反馈循环**:
```
IER-KG提供经验 → Agent使用 → 结果反馈 → 更新经验评分
                     ↓                      ↑
                   定期淘汰低分/低频经验 ←──┘
```

**关键指标**:
- 经验检索命中率
- 经验使用成功率
- 经验新鲜度
- 知识图谱健康度

## 🌐 大循环机制 (The Big Loop)

### 循环架构

```
┌─────────────────────────────────────────────────────────────────┐
│                      大循环 (The Big Loop)                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   Phase 1: 执行 (Execute)                                       │
│   ├─ CodeDev 完成开发任务                                       │
│   ├─ 生成任务报告 (成功率、Bug、效率)                            │
│   └─ 推送至 Feedback Hub                                        │
│                                                                  │
│   Phase 2: 复盘 (Reflect) ←────────────────┐                    │
│   ├─ SRA 分析任务执行过程                    │                    │
│   ├─ 对比预测 vs 实际                        │                    │
│   ├─ 提炼可改进点                            │                    │
│   └─ 生成复盘报告 ──→ 推送至 Feedback Hub   │                    │
│                                                                  │
│   Phase 3: 进化 (Evolve)                     │                    │
│   ├─ SEA 评估复盘建议的系统性价值            │                    │
│   ├─ 决定是否进行代码/工具/流程改进          │                    │
│   ├─ 执行改进并验证                          │                    │
│   └─ 生成改进报告 ──→ 推送至 Feedback Hub ──┘                    │
│                                                                  │
│   Phase 4: 内化 (Internalize)                                   │
│   ├─ 所有经验写入 IER-KG                                        │
│   ├─ 更新经验成功率评分                                         │
│   ├─ 淘汰低价值经验                                             │
│   └─ 周期性固化到 SOUL.md / GROWTH.md                           │
│                                                                  │
│   Phase 5: 验证 (Verify) ──→ 回到 Phase 2                       │
│   ├─ 下次类似任务验证经验有效性                                 │
│   ├─ 对比"预测"vs"实际"                                        │
│   └─ 闭环完成                                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 循环触发条件

| 触发条件 | 启动循环 | 说明 |
|---------|---------|------|
| CodeDev 任务完成 | 自动 | 默认触发完整循环 |
| 用户主动要求复盘 | 从 Phase 2 | 跳过执行阶段 |
| 发现系统性问题 | 从 Phase 3 | 直接进化 |
| 周度定时 | 完整 | 全面回顾 |
| 经验积累到阈值 | 从 Phase 4 | 强制内化 |

### 循环频率

```
小循环 (任务级): 每次 CodeDev 任务后
├─ 触发: CodeDev → SRA (可选)
├─ 耗时: 几分钟到几十分钟
└─ 输出: 单条经验

中循环 (日/周级): 每日/每周
├─ 触发: 日终/周末
├─ 耗时: 几十分钟到几小时
└─ 输出: 多经验整合、原则提炼

大循环 (月/季度级): 月度/季度
├─ 触发: 经验库达到阈值
├─ 耗时: 数小时
└─ 输出: 系统更新、SOUL/GROWTH 修订
```

### 职责
- 接收所有Agent的输出
- 聚合跨Agent的反馈
- 计算全局指标
- 向用户展示状态
- 触发必要的循环

## 🎯 给你的反馈

### 这个设计的核心优势

| 优势 | 说明 |
|------|------|
| **闭环验证** | 不是只记录经验，而是验证经验是否有效 |
| **双向流动** | Agent之间互相反馈，而非单向传递 |
| **分层循环** | 任务级/日周级/月度级，不同粒度 |
| **数据驱动** | 用指标说话，而非主观判断 |
| **用户可控** | 你可以查看仪表盘、调整参数、触发循环 |

### 潜在风险与应对

| 风险 | 应对策略 |
|------|---------|
| **循环过于频繁** | 设置冷却期，同类任务最小间隔 |
| **反馈延迟** | 异步处理 + 优先级队列 |
| **经验冲突** | 版本控制 + 置信度加权 |
| **用户被信息淹没** | 汇总报告 + 关键指标优先 |

### 需要你的决策

1. **循环频率**: 你希望大循环多久运行一次？
   - 选项: 每次任务后 / 每日 / 每周 / 手动触发

2. **用户介入点**: 你希望在哪里介入？
   - 选项: 完全自动 / 复盘后确认 / 进化前审批 / 完全手动

3. **反馈详细度**: 你希望看到多详细的反馈？
   - 选项: 只报异常 / 关键指标 / 完整报告 / 原始数据

4. **优先级**: 如果资源有限，哪个Agent的反馈最重要？
   - 选项: CodeDev (交付质量) / SRA (学习效率) / SEA (系统健康)
