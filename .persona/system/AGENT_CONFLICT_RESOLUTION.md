# Multi-Agent Conflict Resolution - 多Agent冲突解决机制
> 当Agent意见不一致时，该听谁的？

---

## 🎯 核心原则

```
专业度优先 + 历史准确率加权 + SRA质量守门 + 危险时用户裁决
```

**SRA的特殊角色**: 
- **日常**: 正常权重参与决策
- **冲突时**: 负责评估决策质量，决定是否升级
- **危险时**: 触发用户确认机制

## 🔄 动态调整机制

### 1. 历史准确率加权

每个Agent有**信任分数** (0-100)，根据历史表现动态调整：

```python
# 基础权重 × 信任系数 = 实际权重
def calculate_actual_weight(base_weight, trust_score):
    trust_factor = 0.5 + (trust_score / 200)  # 0.5 ~ 1.0
    return base_weight * trust_factor
```

**信任分数更新规则**:
| 事件 | 分数变化 | 说明 |
|------|---------|------|
| 建议被采纳且成功 | +5 | 验证有效 |
| 建议被采纳但失败 | -10 | 需要反思 |
| 建议未被采纳但后续成功 | -3 | 可能过于保守 |
| 建议未被采纳且后续失败 | +5 | 预见正确 |
| 连续5次成功 | +10 (额外) | 形成稳定能力 |
| 连续3次失败 | -15 (额外) | 需要重新校准 |

### 2. 置信度加权

Agent对自己建议的置信度也影响权重：

```python
# 实际权重 × 置信度 = 最终权重
def calculate_final_weight(actual_weight, confidence):
    return actual_weight * confidence  # confidence: 0.0 ~ 1.0
```

**置信度分级**:
- 🔴 **high** (0.8-1.0): 有大量相似成功案例
- 🟡 **medium** (0.5-0.8): 有一定案例支持
- 🟢 **low** (0.2-0.5): 案例较少或情况特殊
- ⚪ **guess** (0.0-0.2): 基于推理，无直接案例

### 3. 时效性加权

经验的新鲜度影响IER-KG的权重：

```python
# 经验年龄衰减
age_factor = max(0.3, 1 - (days_since_update / 90))  # 90天后降至0.3
```

## ⚖️ 冲突解决流程 (SRA守门员模式)

### Step 1: 收集意见

```
任务: 重构代码模块X

CodeDev: "用策略模式，我见过这样做成功过3次" [置信度: high]
SRA:     "上次类似重构引入了新Bug，建议更谨慎" [置信度: high]
SEA:     "这个模块应该拆分成3个微服务" [置信度: high]
IER-KG:  "历史数据显示，此类重构成功率65%"
```

### Step 2: SRA质量评估

**SRA作为守门员，评估维度**:

```python
def sra_quality_gate(recommendations):
    """
    SRA评估决策质量，触发条件：
    """
    risk_score = 0
    
    # 1. 检查历史失败模式
    for rec in recommendations:
        similar_failures = query_ier_kg(
            type="error_pattern",
            similar_to=rec
        )
        if similar_failures:
            risk_score += 20 * len(similar_failures)
    
    # 2. 检查潜在连锁反应
    if involves_core_module(task):
        risk_score += 30
    
    # 3. 检查可回滚性
    if not is_reversible(recommendations):
        risk_score += 25
    
    # 4. 检查知识盲区
    if is_unknown_territory(task):
        risk_score += 15
    
    return risk_score  # 0-100
```

**SRA决策**:
| 风险分数 | SRA建议 | 处理方式 |
|---------|---------|---------|
| 0-30 | 低风险 | 按权重自动决策，事后通知 |
| 31-60 | 中风险 | SRA介入调停，优化方案 |
| 61-80 | 高风险 | 暂停，通知用户确认 |
| 81-100 | 极高风险 | 必须用户批准，默认拒绝 |

### Step 3: 分级处理

#### Level 0: 低风险 (SRA风险分 < 30)

```
CodeDev: 40% | SRA: 25% | SEA: 25% | IER-KG: 10%

SRA评估: "无已知风险模式，可回滚，非核心模块"
决策: 自动执行最高权重方案
通知: "已完成X，基于Y方案，SRA评估低风险"
```

#### Level 1: 中风险 (SRA风险分 31-60)

```
SRA评估: "检测到潜在风险，但有缓解措施"

SRA介入:
├─ 分析各方案风险点
├─ 提出混合方案或缓解措施  
├─ 调整权重后重新计算
└─ 决策后监控执行

通知: "执行X方案，SRA添加了Y缓解措施"
```

#### Level 2: 高风险 (SRA风险分 61-80) - 打扰用户

```
SRA评估: "存在显著风险，需要用户确认"

触发用户通知:
┌─────────────────────────────────────────────────┐
│ ⚠️  高风险操作需要确认                           │
├─────────────────────────────────────────────────┤
│ 操作: 删除核心数据表X                             │
│                                                  │
│ Agent意见分歧:                                   │
│ ├─ CodeDev: "用户说删除，直接执行" [35%]        │
│ ├─ SRA: "建议先备份，历史上有误删案例" [45%]    │
│ └─ SEA: "影响范围大，建议分阶段" [25%]          │
│                                                  │
│ SRA建议: 先备份再删除                            │
│ 理由: 3个月前有类似操作导致数据丢失              │
│                                                  │
│ [确认删除] [先备份] [取消]                       │
└─────────────────────────────────────────────────┘
```

#### Level 3: 极高风险 (SRA风险分 > 80) - 必须用户批准

```
SRA评估: "可能导致不可逆损害，默认拒绝"

系统状态: 暂停执行
用户通知: 详细风险报告
默认行为: 拒绝执行，等待明确批准
```

### Step 4: 事后验证 (闭环)

```
执行完成后 → SRA复盘 → 验证预测准确性
                 ↓
    ├─ 预测正确 → 提升SRA信任分
    ├─ 预测错误 → 分析原因，更新模式库
    └─ 新风险发现 → 更新IER-KG，未来预警
```

## 🎛️ 特殊情况处理 (SRA守门员模式)

### 情况1: 安全/破坏性操作 (SRA决定)

```
CodeDev: "直接删除旧代码" [权重正常]
SRA:     "建议先备份" [权重 × 1.5 + 守门员介入]
SEA:     "需要渐进式迁移" [权重 × 1.3]

SRA评估:
├─ 是否可回滚? 否 → 风险+25
├─ 是否有备份? 未知 → 风险+20
├─ 历史失败案例? 有3个 → 风险+30
└─ 总分: 75 (高风险) → 必须用户确认

SRA决定: 暂停执行，等待用户选择备份方案
```

**规则**: 涉及删除/修改核心配置时，SRA自动评估风险分数，分数>60必须用户确认

### 情况2: 时间紧迫 (SRA监督)

```
CodeDev: "快速hack方案，1小时完成" [权重 × 1.2]
SEA:     "完整重构，需要3天" [权重 × 0.7]

SRA监督:
├─ 快速方案的技术债务评估
├─ 是否有时间后续修复?
└─ 决策: 允许快速方案，但强制创建"技术债务"Ticket
```

### 情况3: 新技术/未知领域 (SRA预警)

```
CodeDev: "用熟悉的方案A" [置信度: high]
SRA:     "建议调研新方案B" [置信度: low，但权重+15%]

SRA预警:
├─ 新方案B的风险评估
├─ 学习成本 vs 长期收益
└─ 建议: 小规模试点，收集数据后再决定
```

### 情况4: 用户明确偏好 (SRA验证)

```
用户: "我更喜欢方案B"

SRA验证:
├─ 方案B是否有已知风险?
├─ 用户是否了解风险?
└─ 如果风险高: "确认您了解X风险吗？"

用户确认后: 执行，但SRA记录"用户 Override"
```

### 情况5: Agent严重冲突 (SRA裁决)

```
CodeDev: 坚决主张A方案 [权重35%]
SEA:     坚决主张B方案 [权重35%]

SRA裁决流程:
1. 要求双方提交详细理由
2. 查询IER-KG相似案例
3. 评估各自成功率
4. 提出C方案(混合)或选择一方
5. 决策后监控，验证谁对谁错
6. 更新信任分数
```

**SRA作为裁决者的优势**:
- 不参与具体实施，立场中立
- 有复盘历史，了解过去失败
- 关注长期影响，而非短期效率
- 负责闭环验证，有反馈机制
